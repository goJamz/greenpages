# Green Pages Engineering Standards and Code Philosophy

## 1. Purpose

This document captures the implementation posture, coding standards, and development philosophy of Green Pages. It exists so that any developer or maintainer can understand not just what the code does, but why it is shaped the way it is and how it should evolve.

The existing baseline docs explain what the product is, what the architecture should be, what the MVP must do, and what delivery model the organization expects. This document explains how the codebase should feel and grow while getting there.

## 2. Core philosophy

### Keep it simple
Do not add abstractions, layers, packages, or infrastructure until there is a concrete reason. "We might need it later" is not a reason. "This file is 500 lines and covers three unrelated behaviors" is a reason.

The goal is not to write the smallest possible amount of code. The goal is to write code that is easy to understand, easy to change, hard to misread, and still capable of growing with the product.

### Build the consumer before the producer
Build the API endpoints and frontend against seed data first. Prove the data model works by using it in a real product. Only then build the ingestion pipeline to populate those tables from real sources. If the model is wrong, it is cheaper to discover that before the ingestion layer exists.

### Build in vertical slices
Build features top to bottom. Do not build the entire database layer, then the entire API layer, then the entire frontend. Prove that data flows from the database through the API for a single domain before moving to the next domain. That is why sections search was built and tested end to end before people search was started.

### Leave room without building the room
Use `cmd/server/` so that `cmd/worker/` can exist later without restructuring. Use `internal/server/` so that `internal/` can hold other packages later. But do not create `cmd/worker/` or additional packages until there is code that belongs there.

### Split by behavior, not by technical layer
When a file grows, split it by what the code does, not by what kind of code it is.

Do this:
- `sections_search.go` — search query, search types, search handler, search helper
- `sections_detail.go` — detail query, detail types, detail handler, detail helper

Do not do this:
- `sections_types.go`
- `sections_queries.go`
- `sections_handlers.go`
- `sections_helpers.go`

The first approach keeps everything about one behavior in one file. The second forces you to jump between four files to understand one feature.

### Do not create packages prematurely
The following package structures are not appropriate right now and should not be created until there is a real, demonstrated need:

- `internal/repository/`
- `internal/services/`
- `internal/handlers/`
- `internal/sections/`
- a generic SQL utility package

These add indirection, navigation overhead, and import complexity. The current `internal/server/` package with behavior-split files is the right level of organization for this stage of the project.

## 3. Abstraction timing rules

A new abstraction should be added only when at least one of these is true:

1. The same logic is clearly repeated in multiple places.
2. The abstraction removes real maintenance pain.
3. It simplifies the code people actually read.
4. It supports an imminent product need.
5. It prevents a well-understood form of future breakage.

A new abstraction should not be added just because:

- it looks more "enterprise,"
- it might be useful later,
- another codebase uses it,
- or it feels architecturally cleaner in theory.

Anti-patterns to avoid early:
- empty interface layers
- repository wrappers that add no value
- config packages introduced before configuration complexity exists
- workers with no current background workload
- Redis with no measured need
- generic search orchestration before core search behavior is proven

## 4. Go code style

### Variable declarations
Declare variables at the top of each function using explicit `var` declarations with inline comments. This makes it immediately clear what every variable in the function is for.

```go
func (applicationServer *Server) handleSectionsSearch(responseWriter http.ResponseWriter, request *http.Request) {
    var rawQuery        string                // Raw q parameter from the URL.
    var normalizedQuery string                // Cleaned search term for database matching.
    var results         []sectionSearchResult // Results from the database search.
    var searchError     error                 // Error returned by the search function.
    var response        sectionSearchResponse // JSON response payload.
```

This is not standard Go style. Standard Go uses short declarations (`:=`) and declares variables at the point of first use. This project chose explicit declarations because they make each function readable as a self-contained unit. A future developer can read the top of any function and immediately understand what it works with.

Do not use `:=` in this project.

### Receiver names
Receiver names on methods use the full struct name (`applicationServer`), not single-letter abbreviations. This is also non-standard Go, but it is consistent with the project's preference for clarity over brevity.

Go linters like `revive` may flag this. That is expected. The team has made a deliberate choice here.

### Response consistency
Every HTTP response path must include an explicit `WriteHeader` call, even for `http.StatusOK` which Go writes by default. This is a team readability decision. Every response path should look the same:

```go
responseWriter.Header().Set("Content-Type", "application/json")
responseWriter.WriteHeader(http.StatusOK)
_ = json.NewEncoder(responseWriter).Encode(payload)
```

The `_ =` on the encoder acknowledges the ignored error intentionally. Do not suppress errors silently. Either return them, log them, or explicitly acknowledge them with `_ =`.

### No dead state
Structs must not contain unused fields. If a variable or field is set but never read, delete it. Dead state misleads the next person who reads the code.

### Struct field comments
Every struct field should have an inline comment explaining what it holds:

```go
type Server struct {
    httpServer *http.Server // Underlying HTTP server with routing and timeouts.
    db         *sql.DB      // Database handle shared across handlers.
}
```

### Function and method comments
Every exported function and method should have a comment. Follow standard Go doc comment conventions: start with the function name.

## 5. Server and routing structure

### Server struct
The `Server` struct in `server.go` holds the `*http.Server` and all shared dependencies (currently `*sql.DB`). Handlers are methods on this struct so they have access to dependencies without globals or closures.

`server.go` is strictly infrastructure: server setup, dependency injection, route registration, and Kubernetes probes. Business logic lives in domain files (`sections_search.go`, `people_detail.go`) that share the same package and attach handlers to the `*Server` struct.

### Route registration
All routes are registered in `registerRoutes()` in `server.go`. This is the single place to see every endpoint the application exposes. Handler implementations live in their respective domain files.

Route registration order matters. More specific routes must be registered before wildcard routes. `GET /api/sections/search` must come before `GET /api/sections/{sectionID}`, otherwise the mux matches "search" as a path parameter. The same applies to `GET /api/people/search` and `GET /api/people/{personID}`.

### Health and readiness
- `GET /api/health` — liveness probe. Returns 200 if the process is alive. Does not check dependencies.
- `GET /api/readyz` — readiness probe. Pings the database. Returns 200 if the database is reachable, 503 if not.

These are designed for Kubernetes liveness and readiness probes. If the database is temporarily down, readiness fails (pod stops receiving traffic) but liveness passes (pod is not killed and restarted).

## 6. Database conventions

### Driver
The project uses `database/sql` with `github.com/jackc/pgx/v5/stdlib` as the driver. This keeps all application code dependent on Go's standard `database/sql` interface, not on pgx-specific types. The driver can be swapped by changing one import. If there is a future demonstrated need for pgx-native features (LISTEN/NOTIFY, COPY, connection pool tuning), the project can migrate to `pgxpool` at that time.

### Migrations
Migrations are currently executed as Postgres init scripts mounted into `/docker-entrypoint-initdb.d/`. This works for early development where the schema changes frequently and the database is regularly wiped.

When the schema stabilizes, the project should switch to a real migration tool (such as `golang-migrate`, `goose`, or `atlas`) that tracks applied migrations and supports forward-only changes in deployed environments.

Migration files follow the naming convention `000001_description.sql`. Schema changes and seed data are kept in separate files. Migrations that create multiple related objects should be wrapped in `BEGIN` / `COMMIT` so they apply atomically.

### SQL queries
SQL queries are declared as package-level `const` strings, not inline in handler functions. This keeps them readable, grep-able, and separated from Go control flow.

Queries should:
- Use `COALESCE` to convert nullable columns to empty strings or zero values at the SQL level, reducing the need for `sql.NullString` in Go code.
- Use `sql.NullInt64` only for genuinely nullable foreign keys returned by LEFT JOINs where 0 would be a misleading value.
- Filter on `is_current = TRUE` for all entity queries.
- Filter on `assignment_status = 'active'` when joining `billet_occupants`.

### Query and handler design
Prefer handler and query shapes that are explicit and easy to trace.

If a two-query approach is clearer than one giant multi-join query, prefer the two-query approach. Person detail uses a clean person query plus an assignment query rather than a single oversized query that repeats person fields across every joined row.

Separate the database query function from the HTTP handler function. `searchSections` handles the query and row scanning. `handleSectionsSearch` handles the HTTP request, calls `searchSections`, and writes the response. This keeps database logic testable independently from HTTP concerns.

### Data integrity
The schema enforces integrity at the database level:
- `occupancy_status` on billets has a CHECK constraint limiting values to `filled`, `vacant`, `unknown`.
- `assignment_status` on billet_occupants has a CHECK constraint limiting values to `active`, `inactive`.
- A partial unique index on `billet_occupants` ensures only one active primary occupant per billet.
- Partial unique indexes on `people` prevent duplicate active records by `dod_id` and `work_email`.
- A unique constraint on `(billet_id, person_id)` in `billet_occupants` prevents duplicate mappings.

### Row iteration
After every `rows.Next()` loop, check `rows.Err()`:

```go
for rows.Next() {
    // scan...
}
queryError = rows.Err()
if queryError != nil {
    return nil, queryError
}
```

This catches errors that occur during iteration but are not surfaced by individual `Scan` calls. This check is mandatory, not optional.

## 7. Canonical data model posture

The core model is:

**Organization → Section → Billet → Person**

This should continue to guide implementation decisions.

The product is not just a people finder. Sections are first-class results. Billets must exist even when unfilled. Person data sits on top of billet data rather than replacing it.

Do not let implementation shortcuts collapse the model into person-only lookup, billet-only lookup, or organization-only lookup. The value of Green Pages comes from keeping these relationships explicit.

## 8. Source-aware without source-heavy

The project should remain source-aware but not prematurely source-heavy. Low-cost future-proofing is good:
- `dod_id` on people
- `normalized_display_name` on people
- `is_current` on all major entities
- `last_refreshed_at` on all major entities
- `source_system` on billet_occupants
- `effective_date` on billet_occupants

But broader source-ingestion systems (staging tables, worker processes, ETL pipelines) should be deferred until product value and canonical model shape are already proven.

The database is the clean room. External Army data from sources like Vantage is messy, lacks standardization, and varies wildly. The API serves only clean, normalized data from canonical PostgreSQL tables. Messy data will be handled by a separate ingestion process that cleans and upserts into canonical tables. That process does not exist yet, and it should not be built until the product loop proves the model.

When integrating external data, identity must be anchored to immutable identifiers like `dod_id`, not to display names which change across data sources.

## 9. Search conventions

### Input normalization
All search inputs are normalized before being used in queries: trim whitespace, lowercase, strip all non-alphanumeric characters. This means "G-6", "G 6", "g6", and "G6" all produce the same search term. This is critical for Army shorthand where the same section can be written many different ways.

The same normalization is applied in SQL using `regexp_replace(LOWER(...), '[^a-z0-9]+', '', 'g')` so that database values match the normalized input. The system should adapt to the user, not force the user to adapt to the system.

### Search ranking
Search results are ranked using a `CASE` expression in `ORDER BY`, not by relevance score. Exact matches rank highest, prefix matches next, then substring matches. This gives predictable, explainable ordering.

### Concatenated matching
Queries support concatenated field matching so that compound searches like "XVIII Airborne Corps G6" work. The SQL concatenates normalized organization name with normalized section code (and other combinations) and checks if the search term appears in the result.

### Deduplication
People search uses `ROW_NUMBER() OVER (PARTITION BY person_id)` to deduplicate results when a person has multiple active billet assignments. Each person appears once in results, with the best-matching assignment shown.

### Response envelope
Search endpoints return a response envelope with the original query and result count:

```json
{
  "query": "G6",
  "count": 4,
  "results": [...]
}
```

### Near-term search posture
Do not rush into a combined mega-search endpoint if separate search endpoints are still clearer during early development. A broader `GET /api/search?q=` can come later when the frontend actually needs it. A broad search that returns the right family of results is more valuable than a theoretically elegant search stack that has not been pressure-tested.

## 10. API conventions

### JSON responses
All endpoints return `Content-Type: application/json`. Error responses use the shape `{"error": "message"}`. Success responses use domain-specific shapes.

### Status codes
- 200 — success
- 400 — bad input (missing parameter, invalid ID)
- 404 — entity not found
- 500 — internal error (database failure, scan failure)
- 503 — service unavailable (readiness check failed)

### Billet status normalization
The API returns billet status with title case (`Filled`, `Vacant`, `Unknown`) regardless of how it is stored in the database. The `normalizeBilletStatus` function handles this conversion. If an occupant exists on a billet, the API overrides the stored status to `Filled` as the occupant data is more trustworthy than a potentially stale status column.

### API design posture
Green Pages APIs should feel predictable. Path names should be obvious, error responses should be consistent, success payloads should be explicit, and one endpoint should answer one main question well.

- Search endpoints return `query`, `count`, and `results`.
- Detail endpoints return one main object and its related data.
- Bad input returns a clear validation error.
- Not found returns a clear not-found error.

## 11. Seed data posture

Seed data is not just test filler. It is currently part of the product-shaping process.

### Rules for seed data
- Seed by business meaning when possible.
- Prefer joins on stable names or natural lookup fields over hard-coded foreign key IDs.
- Deliberately seed realistic scenarios.
- Use seed data to exercise product truth, not just database connectivity.

### Scenarios the seed data should demonstrate
- Filled billets with a single occupant.
- Filled billets with multiple occupants (one primary, one non-primary).
- Vacant billets with no occupant rows.
- Unknown billets with no occupant rows.
- Shorthand-friendly organization and section names.
- Person search tied to real section and organization context.

## 12. Infrastructure decisions

### Container base images
- Builder stage: `golang:1.26.2-bookworm` (Debian-based, official Go support)
- Runtime stage: `debian:bookworm-slim`
- `ca-certificates` installed in runtime for outbound TLS
- Alpine is acceptable but Debian was chosen for full glibc compatibility without requiring `CGO_ENABLED=0`

When deploying to Kubernetes via UDS, the runtime image can be switched to `distroless/static-debian12:nonroot`. During local development, the Debian slim image provides a shell for debugging.

### Docker layer caching
The Dockerfile copies `go.mod` and `go.sum` first, runs `go mod download`, then copies source code. This ensures dependency downloads are cached as a Docker layer and only re-run when dependencies change, not when source code changes.

### Database configuration
The application supports two patterns for database connection configuration:
1. A single `DATABASE_URL` environment variable containing the full connection string.
2. Individual `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`, `DB_SSL_MODE` variables assembled into a connection string at startup.

`DATABASE_URL` is checked first. If set, individual variables are ignored. This supports both local development (where a single URL is convenient) and Kubernetes deployment (where individual secrets may be injected from Azure Key Vault).

### Startup validation
The application pings the database at startup with a 5-second timeout context. If the database is unreachable, the process exits immediately with a clear error message rather than starting and failing on the first request.

### Delivery model
Green Pages follows the organization's delivery pattern: frontend and backend as separate deployable components, container-first development, Kubernetes-targeted deployment, Helm + Zarf + UDS packaging, and ACR as the registry of record. Follow the organization's delivery pattern, but do not copy every service from reference apps by default. Worker is optional. Redis is optional. Extra infrastructure must be justified by actual product need.

## 13. Frontend conventions

### Stack
React 19 + TypeScript + Tailwind CSS v4 + Vite 8.

### Tailwind v4
Tailwind v4 does not use a `tailwind.config.js` file or `postcss.config.js`. Configuration is CSS-first: `@import "tailwindcss"` in the main CSS file and the `@tailwindcss/vite` plugin in `vite.config.ts`. Do not create legacy Tailwind v3 configuration files.

### API proxy
During local development, the Vite dev server proxies `/api` requests to the Go backend at `http://localhost:8080`. This avoids CORS configuration during development. In production, the reverse proxy or ingress handles routing.

### Frontend posture
The first frontend goal is to pressure-test the backend model, not to win architecture points. Do not start with auth wiring, a heavy state library, a large design system, or complex frontend architecture. Start with a search page, a section detail page, a person detail page, direct API fetches, and simple routing.

## 14. What this project avoids

- Premature abstraction
- Service layers without business logic to put in them
- Repository patterns without multiple data sources to abstract
- Interface definitions without multiple implementations
- Dependency injection frameworks
- ORM libraries (the project uses raw SQL with `database/sql`)
- Global state or package-level mutable variables (except compiled regexes)
- Framework-heavy approaches when the standard library is sufficient
- Short assignment operators (`:=`)

## 15. Code review checklist

Before merging code, ask:

- Is `server.go` still strictly infrastructure?
- Are all variables declared at the top of the function with comments?
- Did I explicitly handle or explicitly ignore (`_ =`) all errors?
- Did I write `WriteHeader` on every response path?
- Does every `rows.Next()` loop have a `rows.Err()` check after it?
- Can my SQL query safely handle a null value without crashing the Go scanner?
- Did I split by behavior, not by technical layer?
- If I added an external dependency, did I genuinely need it, or could I have solved the problem with the standard library?
- Does this change make the code easier for the next maintainer to understand?

## 16. When to revisit these decisions

These decisions are right for the current stage of the project: one developer, seed data, no production deployment yet. Revisit when:

- The team grows beyond 2-3 developers and package boundaries would reduce merge conflicts.
- The schema stabilizes and init-script migrations should be replaced with a proper migration tool.
- Production deployment requires `distroless` images, graceful shutdown, or structured logging.
- The ingestion pipeline justifies `cmd/worker/` as a separate binary.
- A real need for connection pool tuning justifies switching from `database/sql` to `pgxpool`.
- Test coverage requires mocking the database, which may justify an interface for the data layer.
- The query and response surface stabilizes enough to justify adopting `sqlc` for type-safe query generation.

---

## 17. Practical test for future decisions

When deciding whether to add a new layer, dependency, abstraction, or service, ask:

1. Does this solve a real current problem?
2. Does this make the code easier for the next maintainer to understand?
3. Does this support the next real product slice?
4. Would the project be worse off in the next few weeks if we did not add it?
5. Are we adding room for growth, or are we building future complexity too early?

If those answers are weak, the feature should wait.

---

## 18. Final takeaway

Green Pages should be built with a calm, deliberate engineering style. It should feel explicit, maintainable, product-driven, readable, lightly future-aware, and resistant to unnecessary complexity.

The project grows by proving one real slice at a time.

That is the intended nature of the codebase.
