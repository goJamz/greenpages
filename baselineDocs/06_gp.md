# Green Pages Engineering Posture and Coding Principles

## 1. Purpose

This document captures the engineering posture, coding standards, and implementation philosophy established during the initial build of Green Pages.

The baseline product and architecture docs explain:
- what Green Pages is,
- what the MVP must do,
- what stack it should use,
- and what delivery model the organization expects.

This document explains **how the codebase should feel and evolve while getting there**.

It exists so future developers and maintainers can understand not just what the code does, but why it is shaped the way it is.

This document supports, and does not replace:
- `01-product-definition.md`
- `02-recommended-architecture.md`
- `03-mvp-requirements-and-data-model.md`
- `04-organization-delivery-and-repo-baseline.md`
- `05-internal-reference-patterns.md`

## 2. Core engineering posture

Green Pages should be built with the following default posture:

- **KISS first**
- **Do not overengineer**
- **Prefer maintainability over cleverness**
- **Prefer readability over abstraction churn**
- **Leave room for growth without building tomorrow’s system today**
- **Push back on weak ideas instead of passively accepting them**

The goal is not to write the smallest possible amount of code.
The goal is to write code that is:
- easy to understand,
- easy to change,
- hard to misread,
- and still capable of growing with the product.

### Practical interpretation

- Do not add layers, packages, or infrastructure because they might be useful later.
- Add them when they solve a real current problem.
- One readable file is better than five abstracted files that hide the flow.
- Prefer explicit code over magic behavior.

## 3. Product-first sequencing

Green Pages should be built by completing real user-facing product slices before building broader infrastructure.

The working rule is:

> **Build the consumer before the producer whenever possible.**

That means:
- finish user-visible directory flows first,
- use seed data to shape the canonical model,
- test the read model under real UI pressure,
- then add heavier ingestion or pipeline machinery later.

### Current example of this rule

The project intentionally chose to build, in order:
- sections search,
- section detail,
- billets,
- people and occupant mapping,
- people search,
- people detail,
- and only then begin the frontend shell.

The project intentionally did **not** jump immediately into:
- Vantage ingestion,
- staging-table frameworks,
- worker services,
- raw JSON landing systems,
- search-document projections,
- or early `sqlc` adoption.

Those things may still come later, but they should arrive after the product loop proves the canonical model.

---

## 4. Canonical model posture

The core model remains:

**Organization → Section → Billet → Person**

This should continue to guide implementation decisions.

Important consequences:
- the product is not just a people finder,
- sections are first-class results,
- billets must exist even when unfilled,
- vacant and unknown are still useful answers,
- and person data should sit on top of billet data rather than replacing it.

### Practical modeling rule

Do not let implementation shortcuts collapse the model into:
- person-only lookup,
- billet-only lookup,
- or organization-only lookup.

The value of Green Pages comes from keeping these relationships explicit.

## 5. Source-aware without source-heavy

The project should remain source-aware, but not prematurely source-heavy.

That means low-cost future-proofing is good.
Examples:
- `dod_id`
- `normalized_display_name`
- `is_current`
- `last_refreshed_at`
- `source_system`
- `effective_date`

But broader source-ingestion systems should be deferred until product value and canonical-model shape are already proven.

### Practical rule

Add the minimum schema or code hooks needed to avoid painting the project into a corner.
Do **not** add a full platform just because you know a future data source exists.

---

## 6. Standard library and simple primitives first

The default implementation bias should be:
- standard library first,
- simple primitives first,
- explicit code first.

### Examples of this posture

- use `database/sql` before reaching for more specialized database abstractions,
- use ordinary handlers before introducing heavy endpoint frameworks,
- use small helper functions before introducing whole packages,
- use explicit SQL before hiding behavior behind large indirection layers.

### Current live interpretation

At the current stage of Green Pages:
- `database/sql` is preferred,
- `pgx` stdlib as the driver is acceptable,
- a full config package is not yet required,
- a repository layer is not yet required,
- and `sqlc` is still the intended future direction, but not required before the current model and API surface settle.

This is not anti-architecture.
It is architecture with timing discipline.

---

## 7. Abstraction timing rules

A new abstraction should usually be added only when at least one of these is true:

1. the same logic is clearly repeated in multiple places,
2. the abstraction removes real maintenance pain,
3. it simplifies the code people actually read,
4. it supports an imminent product need,
5. or it prevents a well-understood form of future breakage.

A new abstraction should usually **not** be added just because:
- it looks more “enterprise,”
- it might be useful later,
- another codebase uses it,
- or it feels architecturally cleaner in theory.

### Anti-patterns to avoid early

- empty interface layers
- repository wrappers that add no value
- config packages introduced before configuration complexity exists
- workers with no current background workload
- Redis with no measured need
- generic search orchestration before core search behavior is proven
- service layers without real business logic to hold
- dependency injection frameworks

## 8. Code organization principles

### Split by behavior, not by technical layer

When a file grows, split it by what the code does, not by what kind of code it is.

Do this:
- `sections_search.go` — search query, search types, search handler, search helper
- `sections_detail.go` — detail query, detail types, detail handler, detail helper
- `people_search.go` — people search behavior
- `people_detail.go` — person detail behavior

Do not do this early:
- `sections_types.go`
- `sections_queries.go`
- `sections_handlers.go`
- `sections_helpers.go`

The first approach keeps everything about one behavior in one place.
The second forces a developer to jump between multiple files to understand one feature.

### Do not create packages prematurely

The following package structures are usually not appropriate yet and should not be created until there is a real, demonstrated need:

- `internal/repository/`
- `internal/services/`
- `internal/handlers/`
- `internal/sections/`
- a generic SQL utility package
- `internal/config/`

These add indirection, navigation overhead, and import complexity. The current `internal/server/` package with behavior-split files is the right level of organization for this stage of the project.

### Leave room without building the room

Use shapes that allow growth later without forcing a future repo rewrite.

Examples:
- `cmd/server/` now so `cmd/worker/` can exist later if it becomes justified
- `internal/server/` now so additional packages can be added later if they become necessary

But do **not** create those additional binaries or packages until there is code that belongs there.

## 9. Go code style

### Variable declarations

Declare variables at the top of each function using explicit `var` declarations with inline comments.

This is not standard Go style.
Standard Go often uses `:=` and declares variables at the point of first use.
Green Pages intentionally chooses the more explicit style because it makes a function readable as a self-contained unit.

### No short declarations

Do **not** use `:=`.

### Naming

- Use thoughtful, descriptive variable names.
- Receiver names should prefer clarity over brevity.
- Avoid single-letter names unless the scope is truly tiny and obvious.

### Comments

- Add comments to top-of-function variable declarations.
- Add comments to struct fields when they improve understanding of API or data shape.
- Comments should explain purpose, intent, contract, or reasoning.
- Comments should not compensate for unnecessarily confusing code.

If the code is hard to read, fix the code first.

### Explicit over implicit

Green Pages prefers explicit behavior when it improves team readability.

Examples:
- explicitly call `WriteHeader(http.StatusOK)` on successful responses,
- explicitly acknowledge intentionally ignored errors with `_ =`,
- explicitly normalize values when API consistency matters.

### No dead state

If a variable, field, or struct member is no longer used, delete it.
Do not let stale fields accumulate.

## 10. Server and routing structure

### Server struct

The `Server` struct should hold shared dependencies such as:
- `*http.Server`
- `*sql.DB`

Handlers should remain methods on `*Server` so dependencies are available without globals.

### `server.go` stays infrastructure-focused

`server.go` should primarily contain:
- server setup,
- dependency injection,
- route registration,
- liveness and readiness handlers,
- and other infrastructure-level wiring.

Business behavior should live in domain files such as:
- `sections_search.go`
- `sections_detail.go`
- `people_search.go`
- `people_detail.go`

### Route registration

All routes should be registered in one obvious place.
More specific routes should come before parameterized routes when matching behavior depends on order.

## 11. Database and SQL standards

### SQL-first posture

The project is SQL-first.
PostgreSQL is the system of record.
The application should prefer explicit SQL over ORM-heavy approaches.

`sqlc` remains the intended direction, but it should arrive when the schema and query surface are stable enough to justify it.

### Query design principles

Prefer query shapes that are:
- explicit,
- easy to test,
- and easy for a future maintainer to trace.

Preferred patterns:
- one clear responsibility per handler,
- one clear response shape per endpoint,
- one or two straightforward queries when that is easier to reason about,
- shared normalization helpers when the formatting rule is truly cross-cutting.

If a two-query approach is clearer than one giant multi-join query, prefer the two-query approach.

### COALESCE first for optional strings

Use `COALESCE(column_name, '')` for optional strings when that keeps the Go side simpler and more readable.
This helps avoid unnecessary `sql.NullString` usage in common API paths.

Use nullable scan types only when nullability is genuinely important to preserve in the Go control flow.

### Clean 404s over giant joins

Avoid oversized LEFT JOIN queries that make it hard to determine whether the base entity exists.
A clean pattern is often:
- one `QueryRow` for the main entity,
- followed by one `Query` for related rows.

If the first query returns `sql.ErrNoRows`, return a clean 404.

### Always check row iteration errors

After every `rows.Next()` loop, check `rows.Err()`.
This is mandatory, not optional.

### Database integrity belongs in the schema too

When a rule is truly a data-integrity rule, prefer to enforce it in the database as well as the application.

Examples already in posture:
- constrained billet status values
- constrained assignment status values
- uniqueness rules around current people identity fields
- one active primary occupant per billet

## 12. Search posture

Search should feel practical before it feels sophisticated.

### Current principle

Support real Army-style search behavior first:
- shorthand,
- normalized input,
- broad section queries,
- whole-section lookup,
- name lookup,
- office symbol lookup,
- and context-aware search.

### Input normalization

All search input should be normalized before matching:
- trim whitespace,
- lowercase,
- strip non-alphanumeric characters.

The same normalization should also be reflected in SQL matching logic.

### Search ranking

Prefer predictable ranking rules over premature search cleverness.
Exact match, prefix match, and sensible substring ranking are often enough early on.

### Deduplication

If a person or other entity can appear multiple times because of multiple related rows, deduplicate intentionally rather than returning noisy duplicate results.

### Search mindset

A broad search that returns the right family of results is often more valuable than a theoretically elegant search stack that has not been pressure-tested.

## 13. API posture

Green Pages APIs should feel predictable and boring in a good way.

### Desired qualities

- path names should be obvious,
- error responses should be consistent,
- success payloads should be explicit,
- and one endpoint should answer one main question well.

### Current shape that should continue

- search endpoints return `query`, `count`, and `results`
- detail endpoints return one main object and its related data
- bad input returns a clear validation error
- not found returns a clear not-found error
- health and readiness remain separate concerns

### Status normalization

If the database stores practical machine values but the API should present friendlier output, normalize once in a shared helper instead of spreading formatting logic across handlers.

### Near-term posture

Do not rush into a combined mega-search endpoint if separate search endpoints are still clearer during early development.
A broader `GET /api/search?q=` can come later when the frontend actually needs it.

---

## 14. Seed data posture

Seed data is not just test filler.
It is currently part of the product-shaping process.

### Rules for seed data

- seed by business meaning when possible,
- prefer joins on stable names or natural lookup fields over hard-coded foreign key IDs,
- deliberately seed realistic scenarios,
- and use seed data to exercise product truth, not just database connectivity.

### Seed scenarios that matter

The seed data should continue to demonstrate:
- filled billets,
- vacant billets,
- unknown billets,
- multiple occupants on one billet,
- one primary occupant,
- shorthand-friendly organization and section queries,
- and person search tied to real section context.

---

## 15. Integration and data reality

### The database is the clean room

External data sources are messy.
The API should serve normalized, clean application data from PostgreSQL, not raw source payloads.

We do **not** write external API calls directly inside web handlers.

If external ingestion is added later, it should happen in a separate refresh or worker path that cleans and loads data into canonical tables.

### Identity anchors matter

Names change. Formats drift. Source data is imperfect.
When integrating external data, identity should be anchored to the most stable identifiers available.

Examples:
- `dod_id` for people where available
- `source_system` for lineage

## 16. Infrastructure and configuration posture

### Current database posture

- keep `database/sql` for now,
- use pgx stdlib as the driver,
- prefer explicit DB environment variables for local and future secret-driven configuration,
- but allow `DATABASE_URL` override where useful.

### Config timing rule

Do **not** introduce a dedicated config package before configuration complexity justifies it.
Small helper functions in `main.go` are acceptable at this stage.

### Containers and delivery

Green Pages should continue to follow the organization’s real delivery model:
- frontend and backend as separate deployable components,
- container-first development,
- local multi-service development,
- Kubernetes-targeted deployment,
- Helm + Zarf + UDS packaging,
- and ACR as the registry of record.

### Important balancing rule

Follow the organization’s delivery pattern, but do not copy every extra service from reference apps by default.

That means:
- frontend and backend are expected,
- worker is optional,
- Redis is optional,
- and extra infrastructure must be justified by actual product need.

## 17. Frontend posture

The frontend should be built as a real, but deliberately lightweight, shell first.

### Current intended stack

- React
- TypeScript
- Vite
- Tailwind CSS

### Current timing rule

Do not start with:
- auth wiring,
- a heavy state library,
- a large design system,
- or complex frontend architecture.

### Preferred first shell

Start with:
- a search page,
- a section detail page,
- a person detail page,
- direct API fetches,
- and simple routing.

The first frontend goal is to pressure-test the backend model, not to win architecture points.

### Auth timing

Keycloak and OIDC remain part of the intended platform direction, but they are not required in the first frontend shell.
Auth should arrive when the basic product loop already works.

## 18. Review and decision posture

Green Pages should not be developed with passive agreement.

### Expected review behavior

- challenge weak assumptions,
- call out stale recommendations,
- prefer the stronger argument when the facts change,
- and revise earlier guidance when the repo has moved forward.

### Important rule

Older recommendations should not override newer repo reality.
The codebase should move forward from the current working state, not from an earlier mental model.

### Pull request checklist

Before opening a pull request, ask:

1. Is this solving a real current problem?
2. Is `server.go` still mostly infrastructure and wiring?
3. Did I explicitly handle or intentionally ignore all errors?
4. Did I add an abstraction because it helps now, or because it sounds architectural?
5. Can the next maintainer understand this flow without jumping through unnecessary layers?
6. Can the SQL and scanning logic handle nullable data safely?
7. Am I preserving the canonical model instead of collapsing it into a shortcut?

## 19. Documentation posture

Important implementation decisions should be written down when they affect:
- code structure,
- sequencing,
- maintainability,
- deployment expectations,
- or future contributor understanding.

This project should prefer a small number of useful documents over a large number of stale ones.

### This means

- update docs when live decisions materially diverge,
- avoid letting baseline docs become misleading,
- and capture the “why” behind code-shaping decisions, not just the end state.

## 20. Current live decisions that should be preserved

The following live decisions are important enough to preserve explicitly:

- keep the backend incremental and product-slice driven,
- do not jump to ingestion platform work yet,
- delay worker or service expansion until justified,
- delay `sqlc` rollout until the current schema and query surface stabilize,
- use seed data to shape and test the canonical model,
- keep `database/sql` for now,
- prefer explicit DB environment variables for local and future secret-driven configuration,
- keep frontend auth for later rather than front-loading it into the first shell,
- normalize API output for consistency,
- keep health and readiness separate,
- and prefer simpler query and handler structures when they are easier to maintain.

### Important doc note

Some older docs still reflect an earlier access interpretation.
The live decision that should be preserved is:
- access is CAC-authenticated,
- limited to users with a `.mil` email associated with the certificate,
- with no RBAC in MVP.

That should not be lost when other docs are updated later.

---

## 21. Practical test for future decisions

When deciding whether to add a new layer, dependency, abstraction, or service, ask:

1. Does this solve a real current problem?
2. Does this make the code easier for the next maintainer to understand?
3. Does this support the next real product slice?
4. Would the project be worse off in the next few weeks if we did **not** add it?
5. Are we adding room for growth, or are we building future complexity too early?

If those answers are weak, the feature should usually wait.

---

## 22. Final takeaway

Green Pages should be built with a calm, deliberate engineering style.

It should feel:
- explicit,
- maintainable,
- product-driven,
- readable,
- lightly future-aware,
- and resistant to unnecessary complexity.

The project should grow by proving one real slice at a time.

That is the intended nature of the codebase.
